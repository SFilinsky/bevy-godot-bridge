## Layers

We break down simulation into two parts:

- Decision (Control) layer
	- this layer can be frame-rate dependent and run on Update
	- Intentions
		- Bevy events, defined alongside Intention Mapping Systems
		- why it should exist:
			- helps to decouple decision making from simulation - we define high-level semantic Events (Intentions) that are available both to the simulation and player control binding, making game actions control-agnostic (Player can be replaced with AI easily and vice versa)
		- examples:
			- EatCreatureIntention
			- BuildStructureIntention
			- AttackCharacterIntention
	- Decision Systems
		- responsibility: generate Intentions for everything that happens in game
	- Intention Nodes
		- responsibility: provide API which can be used on Godot side to produce Intentions, for example when player controls are triggered
	- Intention Mapping Systems
		- responsibility: map Intentions to Mutations, which describe specific changes to game state
		- Why it should exist: 
			- tells what side-effects intention produces, decoupling decision-making from actual state changes
		- examples: 
			- EatCreatureIntention -> ChangeEnergyMutation for eater + RemoveEntityMutation for victim
			- BuildStructureIntention -> SpawnEntityMutation
			- AttackCharacterIntention -> ChangeTargetMutation for attacker

- Simulation layer
	- this layer should be frame-rate independent and run on FixedUpdate
	- Mutations
		- Bevy Events, are defined alongside Components, together with Mutation Systems
		- why is should exist:
			- encapsulating Component mutations - we decide how components can be changed alongside with their definitions, grouping mutation logic and making it easier to track and maintain mutations
			- allowing most systems to run in parallel as they only read components, and writing if offloaded to Mutation Systems
			- Unifying mutations from Intention Mapping and Simulation Systems as one data pipeline
	- Simulation Systems
		- responsibility:
			- produce Mutations on their own, define "laws of nature" in the game that happen independently from player decisions
		- why it should exist:
			- simulating regular events that happen independently of decision making
		- examples:
			- Regenerate 10% energy every turn - ChangeEnergyMutation for every entity with EnergyComponent
			- Gravity - ApplyForceMutation for every PhysicsBodyComponent every tick
	- Mutation Systems
		- responsibility: update component according to Mutations
		- why it should exist:
			- Mutations is package with data, Mutation System takes it and changes Component accordingly

## Data Flow

Data flow per Bevy tick will look like that:

In: Controls/AI → Intentions (events) → Mapping (events) → Mutations (events) → Mutation Systems (write to components)
Out: z


## Reading simulation state

On low level, we can expose Exporter nodes that track state mutations and generate signals (Created, Updated, Removed) for them. This, however, is very low-level as Godot side would need a low of logic to wire it together, especially for use case that require to read state from multiple components.

For that reason, we can introduce higher-level entity-specific wrappers. For each entity we introduce Entity-type tag component, like UnitComponent - it will mark specific types of entities. Then, we track entities with them, and expose higher-level Exporter node that will expose changes from multiple components on Entity. 

For example, if Unit has Position, Health and Energy components - we create wrapper with 
same 3 signals - Created, Updated, Removed - but they return whole set of component DTO instead of one.

This will allow tracking entity as a whole on higher-level, which is much more useful and will require less handling code.

## Tools

To make system easier to use, tools need to be introduced:
- Creating ComponentDTO and Exporter objects
- Creating entity tag components and Exporter objects for entities
- Binding to exporters on Godot side
